# Text Generation 功能文档（中文游戏 UGC）

本文档说明「文本生成」子系统的功能边界、模型与参数、与检测模块的接口，以及实现时的学生向代码约束。

---

## 1. 功能概述

- **做什么**：根据用户给出的中文提示，生成一段 **游戏 UGC 风格** 的中文文本（示例场景：装备描述、技能说明、公会招募、角色对话草稿、世界频道发言草稿等）。
- **不做什么**：不对生成内容做最终「是否违规」的裁定；**裁定由 [03_文本敏感词与有毒内容检测.md](03_文本敏感词与有毒内容检测.md) 中的检测模块完成**。
- **风险提示**：语言模型可能生成不当内容，因此 **生成后必须自动走分类检测**（见总流程文档）。

---

## 2. 技术方案（Hugging Face）

- **任务类型**：`text-generation`（因果语言建模解码）。
- **实现方式**：使用 `transformers` 的 **`pipeline("text-generation", ...)`**，模型与分词器均来自 **Hugging Face Hub**。
- **模型选型说明**：需在实现阶段选定一个在 Hub 上可稳定加载的中文因果 LM（例如小参数量的中文 GPT-2 系列或同类模型）。文档阶段可写「候选列表 + 最终选型以 `README` 或 Notebook 为准」，**不得使用 OpenAI API** 等非课程推荐来源。

---

## 3. 用户可见参数（建议）

| 参数 | 含义 | 说明 |
|------|------|------|
| `prompt` | 用户提示文本 | 必填；需校验非空与最大长度 |
| `max_new_tokens` | 最大新生成 token 数 | 限制长度，防止过长输出 |
| `temperature` | 采样温度 | 值越大越随机；可给默认值与范围说明 |
| `top_p` | 核采样 | 可与 temperature 配合使用 |
| `do_sample` | 是否采样 | 与贪心解码二选一策略需在代码注释中说明 |

**输入安全（可选、简单即可）**：

- 对 `prompt` 做长度截断（避免超长上下文）；
- 不在首版实现复杂「越狱」防护，但可在文档中说明「检测模块为必备防线」。

---

## 4. 输出与接口

**生成函数（逻辑层）建议约定**：

- **输入**：`prompt: str`，以及上述生成参数（可用默认值字典集中管理）。
- **输出**：
  - `generated_text: str`（建议从 pipeline 返回结果中 **显式取出** 主字段，避免把整段 dict 传给下游）；
  - 可选：`elapsed_sec: float`，便于在 UI 或实验中记录延迟。

**交给检测模块的数据**：仅传递 **`generated_text` 字符串**，保持接口简单、便于单元测试与 Notebook 复现。

---

## 5. 与主流程的衔接

顺序固定为：

1. `get_text_generation_pipeline()`（懒加载，首次调用时加载模型）；
2. 调用 pipeline 得到文本；
3. 将文本传入检测模块（词典可选 + `text-classification`）。

详见 [01_总功能流程.md](01_总功能流程.md) 中的架构图与数据流表。

---

## 6. 实现与代码规范对齐

**权威约束**：[执行指南_代码生成规范.md](../执行指南_代码生成规范.md)；风格参考：[DL_NIJIALU_21237096.ipynb](../DL_NIJIALU_21237096.ipynb)。

与本模块相关的摘录：

1. **懒加载**：模块顶部 `_text_generation_pipe = None`，提供 `get_text_generation_pipeline()`，内部判断若为 `None` 则创建 `pipeline`，与参考 Notebook 中多 pipeline 懒加载一致。
2. **英文 docstring**：对 `get_text_generation_pipeline`、`generate_ugc_text`（或同等命名）写清 What、Args、Returns、输入为空等行为。
3. **学生向实现**：解析 pipeline 返回值时用清晰步骤（中间变量），避免深层嵌套推导式；循环控制解码长度时优先可读 for/if，而非炫技式一行写法。
4. **注释**：在「截断 prompt」「从结果取字符串」等处用 1–2 行英文注释说明 **为什么** 这样做。
5. **库范围**：以 `transformers`、`torch` 为主；不引入非必要的异步或复杂框架。

**反例（应避免）**：为生成模块设计多层继承的「通用 LLM 框架」、在首版使用复杂自定义生成循环（除非课程明确要求且能在 docstring 中讲清楚）。
