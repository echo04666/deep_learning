# 文本敏感词与有毒内容检测功能文档

本文档说明如何对 **生成后的中文文本** 进行合规与风险检测：可选 **敏感词规则层** + 必选 **基于 Hugging Face 的文本分类（微调模型）**。

---

## 1. 问题定义


| 层级        | 典型手段                         | 擅长            | 局限                     |
| --------- | ---------------------------- | ------------- | ---------------------- |
| 敏感词 / 违禁词 | 词典匹配、简单规范化后匹配                | 快速、可解释（列出命中词） | 谐音、拆字、上下文反讽难以覆盖；易误杀正常词 |
| 有毒 / 冒犯语义 | 监督学习分类器（本项目的第二个 HF pipeline） | 利用上下文与语义模式    | 依赖数据分布；可能与「游戏黑话」域有偏移   |


**组合策略（推荐）**：

- **词典**：若命中 → 直接标为高风险或「需人工复核」，并展示命中词（实现时用简单循环或少量正则即可）。
- **分类器**：对全部句子运行 `text-classification`，输出逐句 `label` 与 `score`。
- **最终展示**：由业务规则合并（例如：词典命中 **或** 分类分数超过阈值 → 警告）。首版可用单一阈值 + `st.warning` / `st.error` 区分严重程度。

---

## 2. Hug Face 技术方案

- **任务类型**：`text-classification`。
- **模型（强制）**：必须使用你已微调并发布在 Hugging Face Hub 的 `echovivi/CustomModel_bertToxiCN` 作为检测模型，不可替换为其他分类模型。
- **加载示例**：`pipeline("text-classification", model="echovivi/CustomModel_bertToxiCN", tokenizer="echovivi/CustomModel_bertToxiCN")`。
- **输出**：
  - 单标签：取 `argmax` 对应的 `label` 与 `score`；
  - 若为多标签头，需在实现文档中写明阈值策略（学生项目建议优先 **单标签二分类或多分类** 以降低复杂度）。

**与生成模块的接口**：

- **输入**：`text: str`（即生成模块输出的 `generated_text`，进入检测前按句切分）。
- **输出**：建议返回逐句结果列表，例如：
  - `{"sentences":[{"id": str, "text": str, "label": str, "score": float, "dict_hits": list, "is_violation_risk": bool}]}`。
  - 其中 `is_violation_risk` 由业务规则合并得到（如词典命中或分类分数超阈值）。

---

## 3. 游戏 UGC 文本的预处理（建议）

中文游戏场景常见：无分词空格、缩写、谐音、表情符号混用。

- **基础规范化**（保持简单）：去首尾空白、可选全半角统一、过长文本截断至分类器 `max_length`。
- **不在首版强求**：复杂分词、专用游戏词表消歧；若实验中发现误报高，再在报告中作为「局限与改进」讨论。

## 4. 阈值与产品文案（逻辑层）

- 在代码或配置区集中定义常量，例如 `TOXIC_SCORE_THRESHOLD = 0.5`（具体数值由验证集与人工抽检调整）。

## 6. 本阶段功能描述（逐句检测与可编辑复检）

本阶段将检测单元从“整段文本”明确为“句子级别”，便于用户对问题内容做最小化修改。

- **句子拆分**：文本生成完成后，先按句号、问号、感叹号、换行等规则切分为句子列表；每句作为独立检测对象。
- **逐句检测**：每句独立执行词典扫描（可选）与分类检测（必选），逐句返回 `label`、`score`、`dict_hits` 与是否疑似违规。
- **分栏展示**：检测步骤将每句话放入独立栏目/卡片展示，便于用户逐条查看与处理。
- **逐句编辑与重检**：任意句子支持单独编辑并单独再次检测；编辑后仅更新该句检测结果，不影响其他句。
- **删改增操作**：
  - 支持直接删除疑似违规句；
  - 支持新增一句文本后触发该句检测；
  - 支持保留已通过句子并仅处理未通过句子。
- **长度约束**：手动输入（新增或编辑）时，单句最大长度为 `100` 字；超长需阻止提交并提示用户精简。

**交互规则建议（首版）**：

1. 页面提供“单句重检”与“全部重检”两种操作；
2. 单句编辑后默认触发“单句重检”，避免整页重复推理；
3. 全部重检用于批量修改后的最终确认；
4. 删除句子后应立即从检测结果列表中移除，避免展示过期状态。

## 7. 实现与代码规范对齐

**权威约束**：[执行指南_代码生成规范.md](../执行指南_代码生成规范.md)；风格参考：[DL_NIJIALU_21237096.ipynb](../DL_NIJIALU_21237096.ipynb)。

与本模块相关的摘录：

1. **懒加载**：`_toxic_classification_pipe = None` + `get_text_classification_pipeline()`，首次推理再加载微调权重，避免 Streamlit 重复启动时重复占用内存（按实际部署调整）。
2. **词典扫描**：用 **显式 for 循环** 遍历词表列表或使用可读的正则；避免难以阅读的一行链式调用。
3. **分类调用**：封装为短函数，例如 `classify_toxicity(text: str) -> dict`，带英文 docstring，内部调用 pipeline 并 **分步** 取出 `label` / `score`。
4. **模型一致性约束**：Notebook、Streamlit App、报告中的检测模型名称必须统一为 `echovivi/CustomModel_bertToxiCN`，避免评审时出现“训练与部署不一致”。
5. **错误处理**：空字符串、模型路径不存在、pipeline 加载失败时，用 `st.error` 或抛出带说明的异常（与指南一致）。
6. **学生向复杂度**：不构建大型规则引擎或插件化框架；规则与阈值以常量与简单分支为主，便于在报告中解释 Logical Approach。

## 8. 验收与指标（定性 → 定量）

- **定性**：用户能看懂「为何被标为有风险」（词典命中词和/或分类标签与分数）。
- **定量**：分类器本身的准确率、F1、混淆矩阵等必须在微调与实验文档中给出；系统级误报/漏报可通过小规模案例表在报告中讨论（见文档 04）。
