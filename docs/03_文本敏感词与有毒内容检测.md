# 文本敏感词与有毒内容检测功能文档

本文档说明如何对 **生成后的中文文本** 进行合规与风险检测：可选 **敏感词规则层** + 必选 **基于 Hugging Face 的文本分类（微调模型）**。

---

## 1. 问题定义


| 层级        | 典型手段                         | 擅长            | 局限                     |
| --------- | ---------------------------- | ------------- | ---------------------- |
| 敏感词 / 违禁词 | 词典匹配、简单规范化后匹配                | 快速、可解释（列出命中词） | 谐音、拆字、上下文反讽难以覆盖；易误杀正常词 |
| 有毒 / 冒犯语义 | 监督学习分类器（本项目的第二个 HF pipeline） | 利用上下文与语义模式    | 依赖数据分布；可能与「游戏黑话」域有偏移   |


**组合策略（推荐）**：

- **词典**：若命中 → 直接标为高风险或「需人工复核」，并展示命中词（实现时用简单循环或少量正则即可）。
- **分类器**：对 **全部生成文本** 运行 `text-classification`，输出 `label` 与 `score`。
- **最终展示**：由业务规则合并（例如：词典命中 **或** 分类分数超过阈值 → 警告）。首版可用单一阈值 + `st.warning` / `st.error` 区分严重程度。

---

## 2. Hug Face 技术方案

- **任务类型**：`text-classification`。
- **模型**：加载在 **中文有毒 / 冒犯相关数据集** 上微调后的序列分类模型（基座为 Hub 上的中文 BERT/RoBERTa 等，见 [04_分类模型微调与实验计划.md](04_分类模型微调与实验计划.md)）。
- **输出**：
  - 单标签：取 `argmax` 对应的 `label` 与 `score`；
  - 若为多标签头，需在实现文档中写明阈值策略（学生项目建议优先 **单标签二分类或多分类** 以降低复杂度）。

**与生成模块的接口**：

- **输入**：`text: str`（即生成模块输出的 `generated_text`）。
- **输出**：例如 `{"label": str, "score": float, "dict_hits": list}`（词典可选时 `dict_hits` 可为空列表）。

---

## 3. 游戏 UGC 文本的预处理（建议）

中文游戏场景常见：无分词空格、缩写、谐音、表情符号混用。

- **基础规范化**（保持简单）：去首尾空白、可选全半角统一、过长文本截断至分类器 `max_length`。
- **不在首版强求**：复杂分词、专用游戏词表消歧；若实验中发现误报高，再在报告中作为「局限与改进」讨论。

---

## 4. 阈值与产品文案（逻辑层）

- 在代码或配置区集中定义常量，例如 `TOXIC_SCORE_THRESHOLD = 0.5`（具体数值由验证集与人工抽检调整）。
- UI 说明建议区分：
  - **信息性**：展示模型标签与分数，帮助理解；
  - **行动建议**：「建议修改提示词」或「请勿提交该内容」等（与课程演示需求一致即可）。

## 6. 实现与代码规范对齐

**权威约束**：[执行指南_代码生成规范.md](../执行指南_代码生成规范.md)；风格参考：[DL_NIJIALU_21237096.ipynb](../DL_NIJIALU_21237096.ipynb)。

与本模块相关的摘录：

1. **懒加载**：`_toxic_classification_pipe = None` + `get_text_classification_pipeline()`，首次推理再加载微调权重，避免 Streamlit 重复启动时重复占用内存（按实际部署调整）。
2. **词典扫描**：用 **显式 for 循环** 遍历词表列表或使用可读的正则；避免难以阅读的一行链式调用。
3. **分类调用**：封装为短函数，例如 `classify_toxicity(text: str) -> dict`，带英文 docstring，内部调用 pipeline 并 **分步** 取出 `label` / `score`。
4. **错误处理**：空字符串、模型路径不存在、pipeline 加载失败时，用 `st.error` 或抛出带说明的异常（与指南一致）。
5. **学生向复杂度**：不构建大型规则引擎或插件化框架；规则与阈值以常量与简单分支为主，便于在报告中解释 Logical Approach。

## 7. 验收与指标（定性 → 定量）

- **定性**：用户能看懂「为何被标为有风险」（词典命中词和/或分类标签与分数）。
- **定量**：分类器本身的准确率、F1、混淆矩阵等必须在微调与实验文档中给出；系统级误报/漏报可通过小规模案例表在报告中讨论（见文档 04）。
