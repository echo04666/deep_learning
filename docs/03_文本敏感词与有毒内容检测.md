# 文本敏感词与有毒内容检测功能文档

本文档说明如何对 **生成后的中文文本** 进行合规与风险检测：**必选敏感词子串扫描（运行时从 cjh0613/tencent-sensitive-words 拉取词表，内存缓存）** + **必选基于 Hugging Face 的文本分类（微调模型）**。两层须 **同时通过**（逻辑 **AND**）方可进入可发布状态。

---

## 1. 问题定义


| 层级        | 典型手段                         | 擅长            | 局限                     |
| --------- | ---------------------------- | ------------- | ---------------------- |
| 敏感词 / 违禁词 | 词典匹配、子串扫描（GitHub 拉取腾讯系词表，不落盘）                | 快速、与运营清单对齐 | 谐音、拆字、上下文反讽难以覆盖；易误杀正常词 |
| 有毒 / 冒犯语义 | 监督学习分类器（本项目的第二个 HF pipeline） | 利用上下文与语义模式    | 依赖数据分布；可能与「游戏黑话」域有偏移   |


**组合策略（当前实现）**：

- **词表层**：对每句做子串匹配（词表内容来自上游仓库 `sensitive_words_lines.txt`）；若命中则该句 **不通过**（与模型结果合并为统一布尔门控，不在 UI 区分来源）。
- **分类器**：对每句运行 `text-classification`，标签为 `non_toxic` / `toxic`（见代码 `is_predicted_toxic`）；异常标签默认 **拦截**。
- **合并规则**：`is_toxic_final = (模型判定 toxic) or (词表命中)`。**Publish** 要求每句均已检测且 `is_toxic_final` 为假。
- **用户可见**：Step 2 仅展示 **通过 / 不通过** 与统一文案（如「Sensitive content, please modify」）；**不展示**分数、命中词列表，**不说明**是词表还是模型触发。

## 2. Hug Face 技术方案

- **任务类型**：`text-classification`。
- **模型（强制）**：必须使用你已微调并发布在 Hugging Face Hub 的 `echovivi/CustomModel_bertToxiCN` 作为检测模型，不可替换为其他分类模型。
- **加载示例**：`pipeline("text-classification", model="echovivi/CustomModel_bertToxiCN", tokenizer="echovivi/CustomModel_bertToxiCN")`。
- **输出**：
  - 单标签：取 `argmax` 对应的 `label` 与 `score`（`score` 写入会话供内部使用，当前 UI 不展示）。

**敏感词表（强制，在线拉取）**：

- **来源**：运行时从 GitHub raw 下载 [cjh0613/tencent-sensitive-words](https://github.com/cjh0613/tencent-sensitive-words) 的 `sensitive_words_lines.txt`（`main` 分支，**每行一词**；行首 `#` 为注释），**不**在仓库内保存词表文件。
- **加载**：进程内首次检测时 HTTPS 下载，解析后在内存中缓存；详见 [app/data/sensitive_words/README.md](../app/data/sensitive_words/README.md) 与 `toxic_classification_pipeline.load_sensitive_words()`。
- **网络失败或空表**：下载失败、非 UTF-8、或解析后无有效词条时，检测 **报错并中止该次检测**，**禁止静默放行**（与 Streamlit `st.error` / 会话错误状态一致）。

**与生成模块的接口**：

- **输入**：`text: str`（即生成模块输出的 `generated_text`，进入检测前按句切分）。
- **输出（逐句，逻辑结构）**：
  - `label`, `score`（模型）、`dict_hits`（词表命中列表，**默认不在 UI 展示**）、`is_sensitive_hit`（内部）、`is_toxic`（**最终门控**）。

## 3. 游戏 UGC 文本的预处理（建议）

中文游戏场景常见：无分词空格、缩写、谐音、表情符号混用。

- **基础规范化**（当前 v1）：词表扫描前仅 **去首尾空白**；分类器侧由 tokenizer 处理长度。
- **性能说明**：当前实现为对词表逐项子串匹配（`w in text`）。词表极大时可能变慢；若实测成为瓶颈，可改为 Aho-Corasick 等结构（作为后续优化在报告中讨论）。
- **不在首版强求**：复杂分词、专用游戏词表消歧；若实验中发现误报高，再在报告中作为「局限与改进」讨论。

---

## 4. 阈值与产品文案（逻辑层）

- **模型**：以 **标签** 为准（`non_toxic` / `toxic`），不在 UI 展示置信度阈值。
- **词表**：**任意命中**即视为该句未通过（子串匹配）。
- **UI**：统一「未通过」文案；Publish 帮助文案使用 **泛化** 表述（如「automated safety review」），不逐项列举「词表 + 模型」。

## 6. 本阶段功能描述（逐句检测与可编辑复检）

本阶段将检测单元从“整段文本”明确为“句子级别”，便于用户对问题内容做最小化修改。

- **句子拆分**：文本生成完成后，先按句号、问号、感叹号、换行等规则切分为句子列表；每句作为独立检测对象。
- **逐句检测**：每句独立执行 **词表子串扫描**（内存中的拉取结果）与 **HF 分类**；**单句重检 / 全检**均执行完整两层。逐句在会话中保存 `label`、`score`、`dict_hits` 等；**用户界面只反映统一的通过/不通过**。
- **分栏展示**：检测步骤将每句话放入独立栏目展示，便于用户逐条查看与处理。
- **逐句编辑与重检**：任意句子支持单独编辑并单独再次检测；编辑后仅更新该句检测结果，不影响其他句。
- **删改增操作**：
  - 支持直接删除疑似违规句；
  - 支持新增一句文本后触发该句检测；
  - 支持保留已通过句子并仅处理未通过句子。
- **长度约束**：手动输入（新增或编辑）时，单句最大长度为 `100` 字；超长需阻止提交并提示用户精简。

**交互规则建议（首版）**：

1. 页面提供“单句重检”与“全部重检”两种操作；
2. 单句编辑后默认触发“单句重检”，避免整页重复推理；
3. 全部重检用于批量修改后的最终确认；
4. 删除句子后应立即从检测结果列表中移除，避免展示过期状态。

## 7. 实现与代码规范对齐

**权威约束**：[执行指南_代码生成规范.md](../执行指南_代码生成规范.md)；风格参考：[DL_NIJIALU_21237096.ipynb](../DL_NIJIALU_21237096.ipynb)。

与本模块相关的摘录：

1. **懒加载**：`_text_classification_pipe = None` + `get_text_classification_pipeline()`；词表使用 `load_sensitive_words()` 进程内缓存，**首次检测从 GitHub raw 下载**（无本地词表文件）。
2. **词典扫描**：用 **显式 for 循环** 遍历词表列表；见 `scan_tencent_offline_hits`。
3. **分类调用**：`classify_one_sentence`（仅模型）与 `classify_one_sentence_with_wordlist`（模型 + 词表合并）封装在 [app/toxic_classification_pipeline.py](../app/toxic_classification_pipeline.py)。
4. **模型一致性约束**：Notebook、Streamlit App、报告中的检测模型名称必须统一为 `echovivi/CustomModel_bertToxiCN`。
5. **错误处理**：空字符串、词表缺失/空、pipeline 加载失败时，用 `st.error` 或会话错误状态提示（与指南一致）。
6. **学生向复杂度**：不构建大型规则引擎；规则以常量与简单分支为主。

## 8. 验收与指标（定性 → 定量）

- **定性（系统）**：用户可见统一的通过/不通过；不在界面披露子层原因。
- **定性（评审/报告）**：可在报告或内部日志中讨论词表与模型的分工与误报案例。
- **定量**：分类器本身的准确率、F1、混淆矩阵等必须在微调与实验文档中给出；系统级误报/漏报可通过小规模案例表在报告中讨论（见文档 04）。
